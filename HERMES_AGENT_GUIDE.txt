  2.  FILOSOFÍA DE TRABAJO CON HERMES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Basado en tu patrón de trabajo (InmuebleBot ➔ debugging ➔ DB
  migrations ➔ full-stack fixes), estos son los principios clave:

  ─── PRINCIPIO #1: "Resultado, no proceso" ─────────────────────────

  ✓ "Haz el push a git mediante comandos" → Comando listo para copiar
  ✗ "¿Puedes hacer un commit y push?" → Demasiado vago

  Hermes entiende contexto. Dale la meta, no las instrucciones paso
  a paso. Confiá en que va a leer los archivos, entender el proyecto
  y ejecutar la solución óptima.

  ─── PRINCIPIO #2: "Contexto completo desde el vamos" ──────────────

  ✓ "El dashboard add-property está roto después de la migración.
      Log: column properties.lat does not exist. AGENTS.md dice que
      se renombraron columnas durante la migración Frankfurt→Oregon."

  ✗ "El dashboard no funciona, revisá los logs."

  Hermes tiene session_search para recordar conversaciones pasadas,
  pero el contexto actual es más preciso. Incluí el error, el archivo
  relevante y cualquier pista de causa raíz que ya tengas.

  ─── PRINCIPIO #3: "Paralleliza siempre que puedas" ────────────────

  En vez de una sola tarea larga, dividí en sub-tareas paralelas:

  ✓ "Creá 2 tasks en kanban: T1 investiga logs de Render, T2 aplica
      fix y verifica. Máximo 2 sub-workers por task."

  Hermes tiene delegate_task con hasta 5 workers concurrentes y
  kanban con dispatcher automático. Usalos.

  ─── PRINCIPIO #4: "Auditoría después del fix" ─────────────────────

  ✓ "Al finalizar la task, realizar un audit independiente de los
      cambios realizados."

  Siempre pedí una verificación post-fix. Hermes puede compilar los
  archivos, correr tests, y confirmar que los cambios son correctos.

  ─── PRINCIPIO #5: "Documentá en AGENTS.md + skills" ───────────────

  ✓ "También actualizar AGENTS.md con los cambios que realizamos"
  ✓ "Si es un workflow recurrente, guardalo como skill"

  Hermes tiene memoria persistente y sistema de skills. Los fixes
  complejos o workflows que se repiten merecen un skill.

  ─── A. DEBUGGING DE PRODUCCIÓN ─────────────────────────────────────

  Plantilla:
  """
  [error o síntoma]. Analizá los logs y creá una task en kanban
  con [N] workers para fixearlo de raíz. Logs: [pegá los logs aquí]
  """

  Ejemplo real:
  """
  the dashboard is unable to manually add a new property to the db.
  it is broken after db migration, research the problem, propose a
  plan and create 2 task in kanban orquestrator with maximum 2 sub
  workers per task as you see fit to solve the problem and have a
  seamless db migration.
  """

  ─── B. FIXES MULTI-ARCHIVO ────────────────────────────────────────

  Plantilla:
  """
  [Descripción del bug]. Cambios necesarios en [archivos].
  Creá una task con [N] workers. Al finalizar, audit independiente.
  """

  Plantilla:

  """

  8.  KANBAN + SUBAGENTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ─── CUÁNDO USAR KANBAN VS DELEGATE_TASK ─────────────────────────

  Usá KANBAN cuando:
  ✓ Necesitás que el trabajo sobreviva a crashes/restarts
  ✓ Querés human-in-the-loop (revisar antes de continuar)
  ✓ Múltiples especialistas trabajando en paralelo
  ✓ Audit trail importa (todo queda en SQLite)
  ✓ Review/iteration cycle esperado

  Usá DELEGATE_TASK cuando:
  ✓ Tarea rápida (< 5 min) que no necesita persistencia
  ✓ Resultado entra en el contexto sin flood
  ✓ No necesitás seguimiento post-sesión

  ─── PATRÓN: KANBAN + DELEGATE_TASK HÍBRIDO ─────────────────────

  1. Creás la kanban task desde tu sesión
  2. Disparás delegate_task con instrucciones de completar el kanban
  3. El sub-agent ejecuta y hace "hermes kanban complete"
  4. El orquestrador monitorea el progreso

  ─── KANBAN DISPATCHER ───────────────────────────────────────────

  Si querés que los workers se disparen automáticamente:

  hermes kanban daemon

  O si tenés el gateway corriendo, el dispatcher está integrado
  (dispatch_interval_seconds: 60).

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ─── MEMORIA PERSISTENTE ─────────────────────────────────────────

  Hermes guarda facts en dos tipos de memoria:

  • memory(target="user") → perfil del usuario (preferencias, nombre)
  • memory(target="memory") → notas del agente (convenciones, datos)

  SE GUARDA:
  ✓ Preferencias del usuario ("Angelo prefiere respuestas concisas")
  ✓ Hechos sobre el entorno ("Render deploy auto en push a main")
  ✓ Workflows recurrentes (guardar como SKILL)

  NO SE GUARDA:
  ✗ Progreso de tareas (usá session_search para eso)
  ✗ PR numbers, commit SHAs, issue IDs
  ✗ TODO lists temporales

  ─── SKILLS DE PROCEDIMIENTO ─────────────────────────────────────

  Después de una tarea compleja (5+ tool calls) o un fix tricky,
  ofrecé guardarlo como skill:

  "guardá esto como un skill para usarlo después"

  10. TROUBLESHOOTING — PROMTS QUE EVITAN LOOPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ─── PROBLEMA: Hermes se Loop en Tool Calls ───────────────────────

  Solución: Usá la herramienta execute_code para lógica procedural
  en vez de tool calls secuenciales.

  ✓ "Usá execute_code para leer los 3 archivos, compararlos, y
      generar el diff"

  ✗ "Leé el primer archivo, ahora el segundo, ahora comparalos"

  ─── PROBLEMA: El error es el mismo siempre ──────────────────────

  Solución: Después del segundo intento fallido, cambiá de estrategia.
  Pedile a Hermes que investigue primero antes de aplicar fixes.

  ✓ "Analizá la causa raíz primero, proponé un plan, después ejecutá"
  ✓ "Investigá los logs de Render antes de tocar código"

  ─── PROBLEMA: Hermes hace lo que no le pediste ──────────────────

  Solución: Sé explícito sobre lo que NO querés.

  ✓ "Creá una task en kanban, NO ejecutes nada vos mismo"
  ✓ "Solo investigá y reportá, no apliques cambios todavía"

  ─── PROBLEMA: El contexto se pierde en tareas largas ────────────

  Solución: Activaste checkpointing ✓. Usalo.

  "/checkpoint save" → antes de arrancar una tarea compleja
  "/checkpoint rollback <id>" → si algo sale mal

  También: dividí en sub-tareas con delegate_task en vez de una
  sola sesión larga.

  ─── PROBLEMA: El sub-agent no completa el kanban task ───────────

  Solución: Pasá el comando explícito en el context del delegate.

  Context: "Después de fixear, ejecutá:
    cd /path && hermes kanban complete t_xxxx --summary 'Done'"

  12. COMPATIBILIDAD MULTI-LLM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Estos principios funcionan con cualquier modelo (DeepSeek, Claude,
  GPT, Gemini, etc.) porque se basan en el diseño de herramientas de
  Hermes, no en tricks de prompt:

  ─── FUNCIONA SIEMPRE ────────────────────────────────────────────

  ✅ "investigá el error en los logs" (open-ended reasoning)
  ✅ "creá tasks en kanban" (tool calling)
  ✅ "analizá la causa raíz" (reasoning)
  ✅ "reportá los hallazgos" (structured output)
  ✅ "usá execute_code para..." (deterministic execution)
  ✅ "paralelizá con delegate_task" (multi-agent)

  ─── PUEDE VARIAR SEGÚN EL MODELO ────────────────────────────────

  ⚠️ "hacé el push a git" → modelos con tool calling fuerte lo
     hacen mejor (Claude, DeepSeek, GPT-4o)

  ⚠️ "creá un plan detallado" → modelos reasoning (o3, DeepSeek-R1,
     Claude Sonnet) dan planes más profundos

  ⚠️ "sacá conclusiones de estos logs" → funciona en todos, pero
     modelos grandes (70B+) son más precisos

  ─── ESTRATEGIAS QUE NO DEPENDEN DEL MODELO ─────────────────────

  1. Descomponé en pasos pequeños
  2. Usá execute_code para lógica procedural (no depende del LLM)
  3. Validá outputs con herramientas (read_file, terminal)
  4. Skills codifican conocimiento que cualquier modelo puede usar
  5. Session search es cross-model (los datos persisten en SQLite)
  6. La memoria es cross-model (facts sobreviven a cambios de modelo)

  ─── CAMBIAR DE MODELO SOBRE LA MARCHA ──────────────────────────

  /model deepseek-chat      → cambiar temporalmente
  /model claude-sonnet-4    → si tenés Anthropic key configurada
  /model gpt-4o             → si tenés OpenAI key configurada

  hermes config set model.default claude-sonnet-4   → cambio permanente

  14.  /OPTIMIZE_PROMPT — COMANDO PERSONALIZADO
══════════════════════════════════════════════════════════════════════

  Hermes tiene un comando slash personalizado que optimiza cualquier
  prompt automáticamente.

  ─── CÓMO USARLO ─────────────────────────────────────────────────

  1. Simplemente escribí:

     /optimize-prompt <tu tarea o problema>

  2. Hermes:
     a) Lee este archivo (HERMES_AGENT_GUIDE.txt) para entender las
        mejores prácticas de prompteo
     b) Analiza tu requerimiento en profundidad
     c) Decide la mejor estrategia de workers:
        - delegate_task para tareas rápidas y simples
        - kanban para trabajo complejo multi-worker
     d) Crea las tasks optimizadas con la cantidad correcta de workers
     e) Usa los skills disponibles automáticamente:
        - inmueblebot (contexto del proyecto)
        - web-development (UI/frontend)
        - migration (DB/data)
        - browser (navegación web)
        - terminal (builds, deploy, git)
     f) Al finalizar: audit independiente + ofrece guardar como skill

  ─── EJEMPLOS ─────────────────────────────────────────────────────

  ✓ "/optimize-prompt the dashboard add-property is broken. logs say
     column lat does not exist. research and fix it."

     → Crea 2 kanban tasks: análisis de logs + fix de columnas

  ✓ "/optimize-prompt I need to add a new feature: a search filter
     by price range in the dashboard frontend"

     → Crea 3 delegate_task workers: API endpoint → frontend filter
     → integration test

  ✓ "/optimize-prompt recover 9 properties lost in db migration"

     → Crea 2 workers: 1 intenta dump si existe, 1 crea seed script

  ─── REQUISITOS ──────────────────────────────────────────────────

  • Este archivo (HERMES_AGENT_GUIDE.txt) debe estar presente para
    que Hermes lea las recomendaciones de optimización.
  • Los skills relevantes deben estar instalados (todos lo están).
  • Funciona con cualquier LLM (DeepSeek, Claude, GPT, Gemini...).

  ─── COMANDOS RELACIONADOS ────────────────────────────────────────

  /skill optimize_prompt     → Cargar el skill manualmente
  /optimize-prompt <texto>   → Atajo directo (recomendado)
  /reload-skills             → Recargar skills si no aparece
